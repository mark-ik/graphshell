## 1. Product Definition

Canonical prose/UI term: **co-op**. Internal identifiers retain the `Coop` stem in this document.

Co-op is a host-led, real-time co-presence mode for browsing and graph exploration together.

- One host always exists per session.
- The host shares exactly one `GraphViewId` session surface at a time.
- Workbench layout is never shared; each participant keeps their own pane/tile arrangement.
- Cursor presence is shared in real time; cursor style is user-customizable.
- The session UI aims for a game-ish feel: participants are always aware of each other, tooling adapts to context/role.

For persistent communal ownership and shared authority over time, users should use Verse hosting duties rather than co-op.

## 2. Session Identity

Each co-op session has a `CoopSessionId` — a UUID generated by the host at `StartCoopSession` time.

```rust
pub struct CoopSessionId(Uuid);
```

`CoopSessionId` is independent of `GraphViewId`. A single graph view may host multiple sequential sessions over its lifetime. Session identity is used for reducer keying, invite codes, and snapshot attribution.

## 3. Transport

Co-op runs over iroh, reusing the Verso mod's existing iroh `Endpoint`. Co-op is a second ALPN protocol registered alongside Device Sync — it does not require a separate transport stack or mod activation. The Verso mod exposes a `CoopHandle` to the ControlPanel alongside its existing `SyncHandle`.

High-frequency cursor stream events use a dedicated iroh datagram channel (unreliable, low-latency). Durable session events (role changes, contributions, approvals) use iroh streams (reliable, ordered).

## 4. Authority Model

### 4.1 Session roles

- `Host`: owns session policy, shared-node visibility, guest role assignment, approval decisions, and URL/domain policy.
- `Guest(ViewOnly)`: can inspect host-shared nodes; cannot mutate host graph elements.
- `Guest(Contributor)`: can propose graph element create/edit/link operations that require host approval before commit.
- `Guest(Editor)`: can create/edit/link host-shared graph elements without approval.

### 4.2 Non-negotiable authority rules

- There is no neutral session state and no hostless mode.
- Guests can always create personal local nodes/notes that remain private unless they choose to share externally.
- Host authority applies only to the shared co-op surface, not to guests' private workbench/state.

### 4.3 Role escalation mid-session

`SetCoopGuestRole` takes effect immediately when applied. In-flight pending contributions are re-evaluated against the new role:

- `Contributor → Editor`: all pending proposals from that guest are auto-approved on role change.
- `Editor → Contributor` or lower: any further mutations require approval; mutations already committed are not rolled back.
- `any → ViewOnly`: pending proposals are cancelled (returned to proposer as locally-held).

## 5. Privacy and Sharing

### 5.1 Defaults

- First-time share of a graph view starts with all nodes private.
- Host explicitly marks nodes as shared; sharing is allowlist-based.

### 5.2 Reuse of prior sharing profile

Previous node-sharing privacy settings may auto-load only when all joining guests are already in the host's **approved guest list** for that view (a per-view list of previously-approved participants; see §5.4 for deferred scope).

- If a session includes any new/unapproved guest, sharing defaults fall back to private-by-default and require explicit host sharing.
- Host can still load a previous profile manually after reviewing participants.

### 5.3 Sharing granularity

- Share target is node/edge/content element visibility within the hosted `GraphViewId`.
- Shared state does not imply full workspace or full graph export rights.

### 5.4 Approved guest list (deferred)

The "approved guest list" (a persistent list of trusted participants for a given view) is a deferred feature. For now, the host approves guests per-session; no persistence across sessions. Persistent friend/trust lists across sessions are out of scope for the initial co-op rollout.

## 6. Interaction Contract

### 6.1 Presence

- Real-time remote cursor rendering is required.
- Each participant can customize cursor appearance (color/style/name tag) for identification.
- Cursor rendering is presence-only; it does not imply remote input control over another user's desktop/workbench.

### 6.2 Content exchange

- Content exchange path is semantic graph operations: clip notes, create nodes, connect nodes.
- Guests' personal notes/nodes are always available locally even if host disallows contribution to shared nodes.

### 6.3 Approval workflow

- Contributor actions produce pending proposals.
- Host approval commits proposal to shared session state.
- Rejected proposals remain local to the proposer unless explicitly re-submitted.

### 6.4 Pending proposals on session end or disconnect

When the session ends gracefully (`EndCoopSession`) or drops unexpectedly (connection loss):

- Pending proposals (submitted but not yet approved/rejected) are included in the guest's local snapshot, treated as locally-applied.
- Rejected proposals (already rejected by host) are excluded from the snapshot.
- The guest receives a snapshot dialog listing which mutations landed and which are local-only, so they can decide what to keep.

The "review pending mutations" dialog is the minimum acceptable UX; a richer diff review flow is a future upgrade.

## 7. Session Policy Controls

Host config includes session-scoped browsing policy:

- URL/domain allowlist (explicitly allowed domains).
- URL/domain denylist (explicitly blocked domains).
- Precedence: denylist overrides allowlist.
- Policy scope: co-op session only; does not mutate global browser preferences.

## 8. Session End and Snapshot

At session end, each guest can take a local snapshot:

- Snapshot content = host public/shared nodes visible to that guest + guest's own notes/nodes + locally-held pending proposals (see §6.4).
- Snapshot destination = new local `GraphViewId` under guest ownership.
- Snapshot does not transfer host-private nodes or rejected proposals.

## 9. Session UI

Co-op has a lightweight, always-visible presence overlay active whenever a session is running. The goal is a game-like ambient awareness — participants always know who else is present and what role they hold.

### 9.1 Presence overlay (minimum)

- Participant list: avatar/name tag + role badge for each connected participant.
- Cursor presence rendered in graph view for all participants.
- Session status indicator (connected / reconnecting / ended).
- Quick-access to `EndCoopSession` / `LeaveCoopSession`.

### 9.2 Role-sensitive tooling

The command palette and contextual menus adapt to the guest's role within the active co-op session:

- `ViewOnly`: mutation commands hidden or disabled; inspection/navigation commands available.
- `Contributor`: mutation commands trigger the proposal flow rather than direct commit.
- `Editor`: full mutation commands available; no approval step.
- `Host`: full commands + host-only controls (guest role assignment, sharing, domain policy, end session).

Role context is injected into the command palette as a filter/modifier — no separate "co-op mode" command set needed.

### 9.3 Deferred UI features

The following are valid aspirational features, noted for future planning:

- Minichat: in-session text chat panel.
- Voice chat: low-latency audio channel between participants.
- Web clipper UI integrated with co-op proposal flow.
- Richer session history/timeline view.

## 10. Relationship to Device Sync and Verse

- `Device Sync`: replicates durable state between trusted devices for the same owner/workspace set. Not a live co-presence UX.
- `Co-op`: live host-led co-presence with explicit sharing and session authority.
- `Verse`: persistent communal/shared hosting model; may rotate host responsibilities over time.

## 11. Intent Surface

### 11.1 Reducer-owned (current bridge carrier: `GraphIntent`)

This section uses the current reducer bridge carrier naming. The long-term architecture may introduce `AppCommand` / planner layers above it, but durable co-op session mutations still belong on the reducer-owned side of the authority boundary.

These mutate durable co-op session state and belong in `apply_intents()`:

- `StartCoopSession { graph_view_id, host_id, session_id: CoopSessionId }`
- `JoinCoopSession { session_id, guest_id }`
- `LeaveCoopSession { session_id, participant_id }`
- `EndCoopSession { session_id }`
- `SetCoopGuestRole { session_id, guest_id, role }` — re-evaluates in-flight proposals per §4.3
- `SetCoopShareVisibility { session_id, node_id, shared }`
- `SetCoopDomainPolicy { session_id, allowlist, denylist }`
- `SubmitCoopContribution { session_id, contribution }`
- `ApproveCoopContribution { session_id, contribution_id }`
- `RejectCoopContribution { session_id, contribution_id }`
- `TakeCoopSnapshot { session_id }`
- `LoadCoopShareProfile { session_id, profile_id }`

### 11.2 Workbench-owned (`WorkbenchIntent` / GUI intercept)

These mutate pane/tile layout only and must be intercepted before reducer:

- `OpenCoopPanel`
- `CloseCoopPanel`
- `OpenCoopInviteDialog`
- `OpenCoopPolicyDialog`
- `FocusCoopParticipantList`
- `FocusCoopContributionQueue`

### 11.3 Signal/channel-owned (ephemeral presence)

These are non-durable and should not enter undo/WAL. Transmitted over iroh datagram channel:

- Cursor position stream
- Cursor style change broadcast
- Presence heartbeat / latency ping
- Transient selection hover hints

If these events need diagnostics, emit diagnostics directly from the presence channel handler, not via persisted reducer history.

### 11.4 Hard boundary rules

- Workbench intents that leak into reducer path are invariant violations and should warn/panic in debug tests.
- Cursor stream events must never create undo checkpoints.
- `Contributor` proposals become durable only on host approval.
- Domain policy is session-scoped and must not mutate global preferences.

## 12. Guard Tests (Minimum)

- `coop_workbench_intents_are_intercepted_before_reducer`
- `coop_cursor_stream_does_not_touch_undo_or_wal`
- `coop_contributor_mutation_requires_host_approval`
- `coop_editor_mutation_commits_without_approval`
- `coop_first_time_share_starts_private_by_default`
- `coop_reuse_profile_requires_all_guests_preapproved`

## 13. Rollout Order

1. Add `CoopSessionId` type + Coop reducer model + intent enum stubs + no-op arms.
2. Add workbench intercept path + minimal presence overlay shell (participant list, session status).
3. Add presence channel (cursor broadcast) over iroh datagram — diagnostics wired.
4. Implement role gating (ViewOnly/Contributor/Editor) + role-sensitive command palette filtering.
5. Implement snapshot export, domain-policy enforcement, pending-proposal snapshot behavior.
6. Close guard tests, then wire full UI flows.
7. (Deferred) Minichat, voice, richer web clipper integration.

## 14. Flock Model (Deferred)

A **flock** is a peer's full interaction history — every participant encountered across all Coop sessions. Flock membership is automatic: when a session ends, all participants are added to each other's flock. No explicit friend request is needed.

### 14.1 Data model

```rust
pub struct FlockEntry {
    user_id: NostrPubkey,       // cross-device identity anchor (see §15)
    display_name: String,       // last-seen name from session presence
    tags: Vec<FlockTag>,
    known_devices: Vec<NodeId>, // iroh NodeIds seen for this user
    first_seen: SystemTime,
    last_seen: SystemTime,
}

pub enum FlockTag {
    Friend,
    Acquaintance,
    Blocked,
    Custom(String),             // user-defined labels
}
```

`FlockTag::Blocked` is a hard signal: blocked peers cannot join sessions you host and are never auto-approved.

### 14.2 Relationship to session approval

The §5.2 "approved guest list" check maps directly onto flock state: a guest is "previously approved" if they appear in your flock and are not tagged `Blocked`. No separate approval store is needed.

### 14.3 Scope note

Flock is a local, personal data structure — it is not shared or synchronized via Device Sync or Coop. It lives in the local profile store alongside other user preferences.

### 14.4 Rollout note

Flock is not required for the initial Coop rollout. Initial sessions treat every guest as new (§5.1 private-by-default). Flock is added as a follow-on feature; the §5.2 auto-load path is gated behind its presence.

## 15. Peer Identity Model (Deferred)

### 15.1 Two-layer identity

Graphshell uses two identity layers with distinct roles:

| Layer | Type | Scope | Stability |
| --- | --- | --- | --- |
| **User identity** | `NostrPubkey` (`npub`) | Cross-device, shareable | Permanent (keypair-bound) |
| **Device identity** | `NodeId` (iroh) | Per-device transport | Session/device lifetime |

A single user has one `npub` and potentially many `NodeId`s (one per device). The `npub` is the anchor for flock, session invites, and role assignment. The `NodeId` is the anchor for iroh connections and Device Sync version vectors.

### 15.2 Why Nostr

- Identity is a keypair the user owns — portable across phone, desktop, and thin client with no server dependency.
- `npub` (bech32-encoded public key) is a human-shareable identifier suitable for invite codes and QR codes.
- Mobile signing delegation: on Android, the Amber app acts as a signing delegate — the thin client requests a signature without ever holding the `nsec`. The same pattern works on desktop via browser extensions (Alby, nos2x).
- The Nostr relay infrastructure and social layer are optional — we use only the keypair format and NIP-01 signing conventions. No relay dependency.
- If a public discovery or social dimension is added later (sharing graph views, public profiles), Nostr infrastructure is already available without architectural change.

### 15.3 Presence broadcast

Each session presence broadcast includes:

```rust
pub struct PresenceIdentity {
    npub: NostrPubkey,
    device_node_id: NodeId,
    display_name: String,
    /// NIP-01 signature over (session_id || device_node_id || display_name)
    /// Allows peers to verify this device is authorized to speak for this npub.
    signature: NostrSignature,
}
```

Peers verify the signature on join. An unverifiable presence broadcast is rejected. This prevents a device from claiming another user's `npub`.

### 15.4 Multi-device same-user in a session

Two devices belonging to the same user (same `npub`) may join the same Coop session. They appear as a single participant in the flock/participant list (merged by `npub`), with both cursors rendered but attributed to one identity. Host role assignment applies to the `npub`, not the `NodeId` — both devices share the same role.

### 15.5 Relationship to Device Sync identity

The Verso Device Sync pairing ceremony establishes trust between `NodeId`s belonging to the same user. That trust is separate from the Nostr `npub` — Device Sync does not require Nostr. The `npub` layer is additive: if present, it provides cross-device user identity for Coop. If absent (user has not configured a Nostr key), the system falls back to `NodeId`-per-device identity (pre-§15 behavior, every device is a distinct peer).

### 15.6 DID support (deferred)

DIDs (Decentralized Identifiers, W3C) are a natural extension point. `did:key` is structurally equivalent to a Nostr `npub` (just an encoded public key) and could be supported alongside Nostr with minimal extra work. `did:plc` (Bluesky/AT Protocol) and `did:web` add resolvability and open potential AT Protocol integration. Implementation approach: define a `UserIdentity` enum in the identity layer rather than hardcoding `NostrPubkey`, so DID variants can be added without breaking the flock data model.

```rust
pub enum UserIdentity {
    Nostr(NostrPubkey),
    DidKey(DidKeyIdentity),     // deferred
    DidPlc(DidPlcIdentity),     // deferred — AT Protocol integration
}
```

`FlockEntry.user_id` and `PresenceIdentity.npub` should be typed as `UserIdentity` when DID support is added.

### 15.7 Rollout note

Nostr identity is deferred — not required for initial Coop or Device Sync. Initial builds use `NodeId` as the sole peer identifier. Nostr identity is added as a follow-on when the thin client is prioritized.

## 16. Wallet Integration (Deferred)

### 16.1 Concept

A wallet is a personal data and identity vault the user physically owns. Graphshell targets two distinct wallet roles:

1. **Signing device** — wallet holds the user's `UserIdentity` keypair and signs presence broadcasts, session invites, and graph export blobs. No data stored on the wallet.
2. **Data transport** — wallet (or wallet-adjacent infrastructure) carries an encrypted snapshot of the user's graph/workspace state. Plug in (or authenticate) on any device and your graph loads.

These are independent features on different timelines.

### 16.2 Wallet as signing device (near-term)

Hardware wallets (Ledger, Trezor) and software wallets with external signer support can hold the ed25519 or secp256k1 key that backs the user's `npub` or `UserIdentity`. The signing flow:

- Graphshell constructs the payload to sign (e.g. `session_id || device_node_id || display_name`).
- Wallet signs it via its signer protocol (USB HID, NFC, BLE, or browser extension bridge).
- Graphshell receives the signature and includes it in `PresenceIdentity`.

This requires no data storage on the wallet and no changes to the graph data model. It is the simplest wallet integration and should be the first target.

### 16.3 Graph state as wallet-transported data (medium-term)

The goal: carry your browsing graph in your wallet so it is available on any device you authenticate to, without requiring a cloud server.

**Practical approach — encrypted relay export (Nostr NIP-96 / NIP-44):**

- Graph workspace state is serialized and encrypted with a key derived from the user's `npub` (NIP-44 encryption).
- The encrypted blob is posted to one or more Nostr relays the user controls or trusts.
- On any new device, the user authenticates with their key, the client fetches and decrypts the blob from the relay, and the workspace loads.

This is "wallet-transported" in the sense that the key is the passport — the relay is the storage medium, the wallet is the gatekeeper. No hardware storage required.

**Encrypted blob format:**

```rust
pub struct WalletExportBlob {
    schema_version: u32,
    workspace_id: WorkspaceId,
    /// NIP-44 encrypted serialized workspace snapshot
    encrypted_payload: Vec<u8>,
    /// Signing key identity of the exporting user
    author: UserIdentity,
    /// NIP-01 signature over (schema_version || workspace_id || encrypted_payload hash)
    signature: UserIdentitySignature,
    exported_at: SystemTime,
}
```

### 16.4 Hardware data vault (long-term, aspirational)

Some hardware devices (specialized USB drives, secure elements with storage) could hold an encrypted graph snapshot directly — no relay needed. This is the literal "browsing history in a wallet" vision. Deferred until a viable hardware target and storage format emerge. The `WalletExportBlob` format defined in §16.3 is the intended wire format for this path too.

### 16.5 Relationship to Device Sync

Wallet export is complementary to, not a replacement for, Device Sync:

- **Device Sync** (iroh, CP4): continuous real-time replication between trusted devices you own. Requires devices to be online simultaneously or use a relay node.
- **Wallet export**: point-in-time encrypted snapshot, portable to any device via key authentication. No peer device needed.

A user might use both: Device Sync for live continuity between their main machines, wallet export for cold backup or bootstrapping a new device.

### 16.6 Rollout note

Wallet signing (§16.2) is the only near-term target and depends on §15 (Nostr identity) being in place. Relay export (§16.3) and hardware vault (§16.4) are follow-on features with no hard dependency on initial Coop rollout.
